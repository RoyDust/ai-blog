# 后台首页信息层级重构设计方案

- 日期：2026-06-11
- 状态：已实施
- 来源：impeccable 评审（2026-06-08）P1-3 "后台信息层级过载，卡片与标签权重太平均"
- 范围：`src/app/admin/page.tsx` 及其子组件；数据层（`lib/admin-stats` 等）尽量复用，不新增表

## 1. 现状诊断

后台首页当前以相近权重的卡片并列呈现：访问趋势、互动趋势、草稿队列、待审评论、热门文章、AI 模型清单、各类统计徽章。管理员打开首页时无法一眼回答三个问题：

1. **今天有什么必须处理？**（待审评论、失败的 AI 任务、滞留草稿）
2. **内容生产健康吗？**（发布节奏、阅读/互动是否异常）
3. **趋势怎么样？**（流量、热门内容——低频、可下钻）

现有数据获取已齐备：`getDraftQueue()`、`getPendingCommentQueue()`、`getPopularPosts()`、`getDashboardStats()`、`getPublicAiModelOptions()`（见 `admin/page.tsx`），缺的是**层级**而不是数据。

## 2. 目标信息架构：三层结构

```
┌ L1 今日待办（行动层，最高权重）────────────────┐
│ 待审评论 N · 失败 AI 任务 N · 滞留草稿 N · 待发 Newsletter │
│ 每项 = 数字 + 直达链接；全部为 0 时整层收起为一行"今日无待办" │
├ L2 内容健康（指标层，4 个核心数）──────────────┤
│ 7 日发布数 · 7 日阅读时长 · 互动率 · 订阅净增          │
│ 纯数字行，无卡片边框，tabular-nums，环比小箭头          │
├ L3 趋势与明细（下钻层，折叠/次级权重）─────────┤
│ 访问趋势图 + 互动趋势图（保留现状图表）               │
│ 热门文章 Top5 / 草稿队列 / 待审评论详情（从 L1 点击进入锚点）│
│ AI 模型清单 → 移出首页，归入 /admin/ai（首页只留异常提示） │
└────────────────────────────────────────┘
```

权重规则（解决"全是卡片"）：

- L1 是唯一允许使用品牌色强调的区域；徽章只表达状态（红=需处理），不做装饰。
- L2 不用卡片容器，用 `divide-x` 数字行（参照前台阅读统计的处理）。
- L3 图表与列表降一级标题字号；面板入场动画（MotionReveal）全部移除——工作台要稳定不要漂浮。

## 3. 数据与接口变化

| 需求 | 现状 | 动作 |
|---|---|---|
| 待审评论数/队列 | `getPendingCommentQueue()` 已有 | 复用，补 count |
| 失败 AI 任务数 | `AiTask`/`AiTaskItem` 有状态字段 | `lib/admin-stats` 新增 `countFailedAiTasks(window)` |
| 滞留草稿（>7 天未更新） | `getDraftQueue()` 已有 take 3 | 加 `updatedAt < now-7d` 计数变体 |
| 待发 Newsletter | `NewsletterCampaign` 状态机已有 | 新增 pending 计数查询 |
| L2 四指标 | `getDashboardStats()` 已含大部分 | 缺口在环比（上一窗口对比），扩展现有聚合 |

全部为只读聚合查询，无 schema 变更；新查询进 `lib/admin-stats.ts` 并补单测（项目契约测试风格）。

## 4. 实施批次

| 批次 | 内容 | 验证 |
|---|---|---|
| D1 | `lib/admin-stats` 扩展（失败任务/滞留草稿/待发计数 + 环比），纯函数单测 | vitest |
| D2 | L1 待办条组件（`AdminTodoStrip`）+ 全零收起逻辑 | 组件测试：有/无待办两态 |
| D3 | 首页重排：L1/L2/L3 装配，AI 模型清单迁出，删除装饰性入场动画 | 既有 admin 契约测试 + 截图 |
| D4 | 文案与可达性：每个待办项有 aria-label 与直达链接；键盘走查 | 手测 + axe |

## 5. 验收标准

- [ ] 打开 /admin 首屏（1440×900）能同时看到 L1 全部待办项与 L2 四指标，无需滚动。
- [ ] 所有待办数字可点击直达对应工作台（评论审核、AI 任务、文章列表筛草稿、newsletter）。
- [ ] 全零状态：L1 收起为单行，不留空卡片。
- [ ] 首页品牌色强调元素 ≤ L1 区域 + 1 个主操作按钮（新建文章）。
- [ ] 既有 admin 契约测试全过；新增聚合查询有单测。

## 6. 风险

- "管理员第一任务是什么"是产品判断（评审遗留问题 #2）：本方案押注"处理待办优先于看数据"。若站主更看重数据观察，L1/L2 顺序可对调，结构不变。
- 环比窗口聚合可能慢 → 沿用现有 `getAdminStatsRangeWindow` 缓存路径，必要时单查询合并。

## 7. 工作量预估

D1–D4 合计约 1–1.5 天。建议在《前后台视觉统一》B1–B2 落地后实施，避免同文件双向冲突；若先做本方案，徽章/按钮样式沿用现状 token，不提前引入新视觉。
