## P5 — 分析端点无频率限制

**文件：** `src/app/api/analytics/visit/route.ts`

### 风险

公开 POST 端点，无频率限制。攻击者可无限刷访问量，造成：
- 数据库写入压力
- 文章阅读量数据污染
- 存储空间膨胀（`VisitLog` 表持续增长）

### 修复

在 route handler 开头加入频率限制：

```ts
// 建议 30 req/min per IP
await checkAnalyticsRateLimit(request);
```

同时考虑 `shouldTrackVisitPath()` 是否需要更激进的过滤（如排除明显的扫描路径）。

### 验收标准

- [x] 同一 IP 短时间大量请求返回 429
- [x] 正常浏览不受影响
- [x] 频率限制错误返回明确 JSON 响应

### 修复记录

- `src/lib/rate-limit.ts` 新增 `analyticsLimiter` 和 `checkAnalyticsRateLimit(request)`，限制为每 IP 每分钟 30 次。
- `src/app/api/analytics/visit/route.ts` 在路径过滤后执行限流，超限返回 429。
- 新增回归测试覆盖第 31 次请求被拒绝。

## P6 — 搜索无最大查询长度限制

**文件：** `src/app/api/search/route.ts`

### 风险

只校验了 `query.length >= 2`，没有上限。传入超长字符串可能导致：
- 数据库 ILIKE 扫描极慢，拖垮 PostgreSQL
- AI 搜索模式消耗大量 token

### 修复

在长度校验处添加上限：

```ts
const MAX_QUERY_LENGTH = 200;
if (query.length < 2 || query.length > MAX_QUERY_LENGTH) {
  // 返回 400
}
```

### 验收标准

- [x] 超过 200 字符的查询返回 400
- [x] 正常搜索行为不变

### 修复记录

- `src/app/api/search/route.ts` 新增 `SEARCH_MAX_QUERY_LENGTH = 200`。
- 新增回归测试覆盖 201 字符查询返回 400。
